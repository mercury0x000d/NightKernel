Section 0 - About the Night Kernel

A synopsis of Night

The Night kernel has been designed from the ground up to be a full 32-bit drop-in replacement for the original 16-bit kernel of the FreeDOS operating system, and is intended to be loaded by the existing FreeDOS bootloader. Night is built with GCC C and handcrafted assembly code using the NASM assembler, employs linear memory addressing and operates in Protected Mode on the Intel x86 architecture. The completed kernel will allow users to enjoy modern abilities such as task switching between applications, protected memory, and increased overall performance in a DOS environment while maintaining full compatibility with their existing DOS applications.

Currently, the Night Kernel is a work in progress; most of the code written to this point is providing internal functionality, and as such the kernel is not yet suitable for everyday use. Night will not, at present, run existing DOS applications.



Days before the Night

Although the Night project itself began in June of 2015, its roots go back much further. Its creator originally got the idea to build a custom operating system from scratch after the lackluster release of Microsoft's Windows 95. Initially, this idea manifest as a simple shell with the goal of bringing multitasking and other modern features to older hardware... and 'older hardware' is no joke - the original project featured character graphics and was coded in IBM BASICA on a PC XT!

After a subsequent upgrade to a 80486 machine, the project - then known simply as the Personal Computer Operating System or 'PCOS' for short - evolved as well. The next version was the first to feature color graphics (albeit in only 16 shades), a rudimentary UI and windowing toolkit, and mouse control. It was initially coded with QBASIC but quickly migrated to Visual BASIC as it outgrew the former language. As the project continued, it naturally became desirable to add true preemptive multitasking. Alas, the problem with that is that running multiple simultaneous DOS programs at once, reliably, under a mere shell would be much too complex to be feasible.

After a series of code refinements and feature enhancements pressed the limits of the development rig in use at the time, hardware was upgraded once again, this time to a 667 MHz Celeron machine. The project's source code was migrated to PowerBASIC, and the added processing power of the new PC made possible the incorporation of a virtual machine of sorts which would allow applications compiled for it to run concurrently. The result was a tech demo featuring multiple applications coded in an Assembly language designed especially for the virtual machine, all running under a preemptive scheduler. In time, the project - which had by now taken on the name Exelyx - grew to have not only its own assembler, but a BASIC-to-assembly-bytecode converter as well.

Exelyx was evolved to one last form before being retired permanently in August of 2016; a DPMI-powered MS-DOS program with a code base rewritten in Watcom C. Though Exelyx is dead, some key sections of code developed during its lifespan have now been adapted, in part, for use in a new and refined platform; one which will rely on no underlying operating system or drivers, but shall supply the facilities to manage every aspect of the user's machine completely on its own.

That project is Night.

Section 1 - Building and Running the Kernel

Before we begin, keep in mind these directions are written based on a small variety of Linux-based workstations used by the Night developers. This is not to say, however, that this is the only way to build the kernel; you can adapt these directions to whatever other compatible environment you like. There are no pre-made libraries used in the creation of the Night Kernel; the repo contains all the code necessary to build the kernel from scratch using the NASM assembler without the need to download additional modules.



There are multiple ways to take Night for a test drive:

Booting a pre-made image
The builds folder of the repo is home to a selection of pre-built disk images, each of which are bootable and loaded with the Night kernel. Here you will find a floppy image (NIGHT.IMG), a CD image (NIGHT.iso), and a VDI VirtualBox disk image (Night.vdi); a 16 MiB disk with a base-only install of FreeDOS 1.2 (sans sources) and the default kernel.sys swapped out for the Night Kernel. This disk image is fixed-size to allow for easier mounting under the host operating system and is ready to boot a VirtualBox virtual machine.

To create a virtual machine suitable for testing out Night, the default DOS configuration offered by VirtualBox can, for the most part, be used without alteration, however you should name the machine Night for compatibility with the included makefile.

Note: VirtualBox 6 brought changes over the previous version to how virtual machines are launched from the command line, so the Night makefile has a pair of lines to handle either situation. If you're still using VirtualBox 5.x, simply comment out the default 6.x line and uncomment the 5.x one as noted in the comment inside the makefile itself.



Booting the pre-assembled kernel binary
The repo's builds folder is also home to a pre-assembled kernel.sys, which you can copy over the existing FreeDOS kernel on any bootable FreeDOS medium, then just boot from that bootable media as you normally would.



Building from source
If you're running in a Linux environment and have both VirtualBox and NASM (there were changes in version 2.14 which will not allow Night to compile, so instead use version 2.13.03, available here) installed, you can use the included makefile found in the root of the repo to streamline building and testing. Section 4 - Coding Conventions Used in Night

'C ... all the power of assembly language programming with all the ease of assembly language programming.'
  - Stuart Cheshire, Tricks of the Mac Game Programming Gurus


Assembly has gotten a bad rap as a language both difficult to use and ill-suited for many modern programming projects. Perhaps this is due to Assembly code of days gone by being written in long, rambling blocks with little structure of any kind, or maybe due to certain assemblers of old being rather rigid in the syntax they accepted, but there is no reason why this must be. With a little structure and finesse, Assembly can be forged with no worse readability than comparable C code.

To illustrate this, let's take a look at a function from the Night source code and discuss its structure. Below, we have an example from an older version of the Memory Manager:


  section .text
  MemAllocateAligned:
     ; Returns the address of a memory block of the requested size aligned to the value specified, or zero if unavailable
     ;
     ; input:
     ;   Requesting task number
     ;   Requested memory size in bytes
     ;   Alignment necessary
     ;
     ; output:
     ;   EAX - Address of requested block, or zero if call fails
     ;   EDX - Result code


Every function in Night begins by stating it is part of the .text section of the final binary image, so that the assembler knows to group this function in with the rest of the runnable code. Following this designation is the function label, which in this case is MemAllocateAligned. Immediately after, there is a short description of what the function does, what it expects for input on the stack, what it returns as output, and in what registers that output is returned.




     push ebp
     mov ebp, esp


 Next EBP, the current base pointer, is preserved to the stack and reloaded with the current stack pointer. This provides a reference so that we can more directly access parameters passed to this function and use local variables in the function as well. This is the standard entry procedure for every Night function which accepts input parameters or uses internal variables.




     ; define input parameters
     %define taskNum                      DWord [ebp + 8]
     %define memorySize                   DWord [ebp + 12]
     %define alignment                    DWord [ebp + 16]


 Now that EBP has been preserved and set, offsets from it may be defined to represent parameters passed to this function. Values pushed onto the stack by the caller begin at [EBP + 8] and ascend from there. Since all of Night's functions expect 32-bit parameters, this means that the first parameter passed will live at [EBP + 8], the second at [EBP + 12], the third at [EBP + 16], and so on. As for [EBP + 0] and [EBP + 4], those stack locations are where the saved EBP register and the return address, respectively, live.

Using define directives essentially tells NASM, 'Every time you see memorySize in the code below, go ahead and substitute it with DWord [ebp + 12]'. Note that these directives are not technically required (using a bare reference like DWord [ebp + 12] in the code directly is perfectly valid) but using them makes the code much easier for us humans to read.




     ; allocate local variables
     sub esp, 20
     %define blockAddress                 DWord [ebp - 4]
     %define blockAddressAligned          DWord [ebp - 8]
     %define blockLeadingSize             DWord [ebp - 12]
     %define blockTrailingSize            DWord [ebp - 16]
     %define blockRequestedSize           DWord [ebp - 20]


 Variables local to this function can be represented in a manner similar to that of parameters. To do this, we first must adjust the stack pointer to clear out enough space in the stack area for the variables we will need: in this case 20 bytes, enough space for five double-word values. After that, the individual variables can be defined.

A handy mnemonic for remembering which EBP references are which is that locals are lower than EBP, and parameters are positive.




     ; make sure alignment is 2 or greater to avoid both wasted time (1) and division by zero errors (0)
     cmp alignment, 2
     jb .Exit

     ...


 Next in this function is an example of how passed parameters are accessed. As mentioned earlier, using cmp DWord [ebp + 12], 2 here would be totally acceptable by NASM, but thanks to the defines we set in place earlier, we can make the code much more easy to read. This code performs a simple check on the alignment passed and exits if it is below two.




     ...

     ; set the return values and exit
     mov edx, kErrNone
     mov eax, blockAddress


 And now, skipping ahead to the end, the work of this function is complete. Remember those registers mentioned above in the function description? This is where they get set prior to exiting the function.




     .Exit:
     %undef taskNum
     %undef memorySize
     %undef alignment
     %undef blockAddress
     %undef blockAddressAligned
     %undef blockLeadingSize
     %undef blockTrailingSize
     %undef blockRequestedSize
     mov esp, ebp
     pop ebp
  ret 12


 And finally, it's time to exit!

The series of %undef statements remove from the preprocessor's memory the variable names we %defined earlier. Why is this done? Consider a scenario in which a variable exists in another function later on in the same file, has the same name as a variable in this function, and was accidentally never %defined in its own function. Such a variable name would point to the last existing definition of that name - the one in this function. This essentially would allow functions to alter variables in other functions which, as bad as that is on its own, would be especially nightmarish if one of these functions called the other in a nested fashion. Using %undef prevents this.

The mov and pop lines comprise the standard exit procedure for nearly all functions in the Night kernel. In short, they undo the entry procedure performed at the start of this function and restore the stack to the state it was prior.

The ret 12 instruction will not only return to the caller, but also takes an additional twelve bytes off the stack. Why twelve bytes? The answer lies in the number of input parameters passed to this function. MemAllocateAligned takes three double-word parameters as input, and we can clear out the space they use (in essence popping them all back off the stack in one fell swoop) by specifying the total size at the end of the ret instruction. By the same token, if a function instead takes no parameters, a simple ret could be used since there would be no parameter space on the stack of which to be rid.

Note: If you're using Sublime Text as your code editor of choice, install Nessphoro's NASM x86 Assembly package to get the fancy color coding shown above. :)



Bit Suffixes
Some modules in the kernel, such as screen.asm, have both 16 and 32 bit variants of their functions. In these cases, it is customary to add a "16" to the end of the 16-bit function to denote it is 16 bit code; the 32 bit version has no suffix since it is assumed all code in the kernel is 32 bit unless otherwise specified.

Section 7 - The List Manager

Managing smooth operation of a computer system necessitates the kernel keep track of numerous points of data. To ease that task, a library of code was created to manage arrays of stored data: the List Manager. Browsing the kernel source illustrates just how many items in which the List Manager has a hand in tracking; everything from drives and partitions to the PCI devices contained in the computer. A list, in this context, is simply a series of blocks of data of a constant size (called elements) stored in series, preceded by a header, which contains a handful of 32-bit values which describe characteristics of the list.

Each list begins with a tListInfo header, the structure of which is detailed below.

Table 7.1 - List header format
Offset Name Size Description
0x00 signature DWord A string ('list') identifying the beginning of a List Manager list.
0x04 elementSize DWord Size in bytes of each element in this list.
0x08 elementCount DWord Total number of elements in this list.
0x0C listSize DWord Total size of this list (header and data space) in bytes.
0x10 List data



When is a list too much?

The List Manager is versatile enough to handle data of varying sizes with ease, however handling multi-byte data differs enough from handling boolean data that the List Manager offers a separate structure known as a bitfield, which is simply an array of boolean values. This can be used to store a vast number of true/false values.

Each bitfield begins with a tBitfieldInfo header, the structure of which is detailed below.

Table 7.2 - Bitfield header format
Offset Name Size Description
0x00 signature DWord A string ('bits') identifying the beginning of a List Manager bitfield.
0x04 setCount DWord Number of set bits in this bitfield.
0x08 elementCount DWord Total number of bits in this bitfield.
0x0C listSize DWord Total size of this bitfield (header and data space) in bytes.
0x10 Bitfield data

Section 8 - Memory Management in Night

Memory allocation is an important part of any kernel, and Night is no different.


Physical memory

Upon boot, Night probes the BIOS memory map using function 0xE820 of interrupt 0x15. After obtaining these results, the lowest block of RAM found in the map which is both at/above the 1 MiB mark and large enough to do so will be selected to hold the memory management information. The size of this memory management information area will vary based on the amount of memory in the system, but will always hold three things: two bitfields and and a copy of the BIOS memory map itself. The two bitfields will always be equal to each other in size, as they both contain one bit per page of memory in the system; the first is used to track allocated memory pages, and the second tracks which pages are reserved.


Paging considerations

Night uses the PAE feature of x86 processors to enable addressing up to 64 GiB of memory.







Section 9 - Multitasking, Night Style

One of the main benefits Night brings to the DOS experience is multitasking. Some older operating systems common to the DOS era would use a technique called Cooperative Multitasking, which relied on programs to voluntarily yield their control of the CPU back to the operating system before another program could get time to do its job. This is not the way Night does things.

Preemptive Multitasking is a system by which task switches are initiated by an outside source, usually a hardware interrupt provided to the CPU, which means applications cannot monopolize the processor. This is the technique used by Night.



An Overview of Tasking on Night

The Programmable Interval Timer is set to fire 256 times per second. Its interrupt handler simply adds one to the tSystem.ticksSinceBoot value, then checks if the global multitasking switch, tSystem.taskingEnable, is non-zero. If it is, task-switching logic is performed to determine the next valid task to be run in a round-robin manner.

In computing, as in life, certain tasks are sometimes more crucial to complete than others. Extending this principle to the digital workspace, Night's task management structure contains a priority value which dictates how many turns at the CPU the task will have in a row before being switched out for the next waiting task. Currently the priority levels supported range from 0 (one turn before switch) to 255 (an over-the-top value which is mainly there simply for future-proofing. One caveat exists: since the PIT interrupts fire 256 times per second, maxing out the priority value to 255 can cause near-total lockup of other running tasks. Therefore, it is not recommended to use a priority value over 3 so that other tasks do not visibly suffer from lost CPU time. A future version will incorporate a global execution cap, preventing tasks from being set to too high of a priority.

When a task is switched in, Night reads the CPU's Time Stamp Counter and stores the result in switchInLow and switchInHigh. When it is switched back out, the TSC is read once again and the original switch-in values are subtracted from this reading. The resulting value is stored in cycleCountLow and cycleCountHigh and can be used as an estimated number of cycles the task spent doing work during its turn at the CPU. This information can be used by Task Manager-type programs to easily calculate how well tasks are utilizing the time slices they are allocated.

Although the x86 architecture does provide a method of performing multitasking in hardware, Night - like many other operating systems - does not use it. Instead, the faster software method is used via a single Task State Segment which is updated at every task switch.

Night tracks task data using a list of tTaskInfo structures, one per task, each 128 bytes in length.

Occasionally, the kernel will need to determine whether or not a specific slot in the task list is currently in use by a task. The official way of doing this is to examine the kernelStackAddress field of the task's slot in the tasks list; if it is zero, the slot is not in use.



Automatic task focus changing
Each running task has the option of specifying a certain key combination it would like to use to regain focus. Once this specification has been made, the task's tTaskInfo record will contain the scan code of the key to reactivate the task in the magicKey field and the state of the modifier keys in the magicFlags field. This works even if the task is paused at the time of the key press. In fact, the focus switching code built into Night uses this exact technique to trap presses of the Alt-Tab key combination and immediately jump into focus to display the task changing interface.


For added security, however, there are some restrictions on this practice.

  1. A task may not trap simple letter, number, or symbol keys alone - the key must be accompanied by a modifier. That is, Shift + R can be trapped, but R by itself is disallowed.
  2. A task cannot trap a key combination already in use by an existing task - trapping of the key combination must be canceled by the existing task before another task can use the same combination. This prevents newer tasks from "hijacking" the key combinations of any existing tasks.
  3. Any successfully trapped key combination will not appear in any event queue - neither the task losing focus nor the task gaining focus will register it as a key press. The task gaining focus, however, will receive a special type of System event to let it know it has not only just gained focus, but has done so as a direct result of a key trap.

These events are used to provide both a way for applications to stay informed of the system landscape in which they exist, and also a mechanism for inter-process communication.

In the first scenario, Night will send an application a System event as certain significant things occur, such as when an application has just been given focus, loses focus, needs to exit, a menu item has been selected, or even when a new disk becomes available. This helps applications stay on top of what's going on in the system as they run.

The second - and arguably cooler - scenario allows applications to issue commands to other applications. Suppose you're a fancy new GUI, and your user wants to print a document but doesn't need any of the other dozens of features offered by the creating application. What do you do? Instead of making the user open the file, wait for the creating application to load, issue a Print command, and finally close the application, you can simply load the application and pass to it a System event letting it know that it's merely needed to print this file and exit. The application can then take care of the rest, perhaps even not going to all the trouble of loading all its resources since it knows it's only needed for a quick job, should it be coded with such intelligence. Another possibility - what if a user has a series of operations they need repeated over and over again with precision? A driver can be made which will dip into the event stream of another application and record what it finds there. This allows for some very cool scripting possibilities in which a user can thread together a flow of events across multiple programs at the kernel level to get work done in fully automated fashion.



Whose event is it, anyway?
For security, only the application which currently holds focus will receive all events, and those running in the background will only receive System events. Night-aware applications can, however, indicate they wish to be notified when a certain key is pressed, allowing them to jump into action from the background when a required key combination has been made. Traditional DOS applications never receive events directly, per se, but they will be able to get the data from mouse and key events by using their typical means with Night handling the translation for them automatically as explained further below. Traditional DOS programs will never receive System events, as there is no common established interface for this data to be passed to them.



Events and DOS applications
Although traditional DOS applications will not be aware of the added functionality of this system of events, they can still benefit from them. Just like modern Night-aware apps, Night will set up a full event queue for traditional DOS programs even though they won't directly realize it. Any time they use typical methods of polling for input, however, - such as checking port 0x60, for example, or calling a mouse interrupt - the input they receive will actually be pulled from the data in their event queue instead of directly reading the hardware, and their scriptability potential will still be viable.
